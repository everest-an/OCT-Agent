# 02 · File Layout — Mission 持久化结构

> **更新日期**：2026-04-17
> **前置阅读**：[01-DESIGN.md](./01-DESIGN.md)

## 一、根路径

```
~/.awarenessclaw/
├── missions/
│   ├── <mission-id>/        # 每个 mission 一个目录
│   │   ├── mission.json
│   │   ├── plan.md
│   │   ├── MEMORY.md
│   │   ├── HEARTBEAT.md
│   │   ├── artifacts/
│   │   └── logs/
│   └── ...
└── orchestrator.pid         # S2 引入：Runner daemon 的 PID 文件
```

**选 `~/.awarenessclaw/` 的理由**：
- 独立命名，不和 OpenClaw 的 `~/.openclaw/` 混
- 和现有 `~/.awareness-claw/openclaw-runtime/` 错开（那是旧 runtime 路径）
- 所有 mission 在一个顶层目录，便于备份 / 删除 / 清理

**跨平台**：
- macOS / Linux：`$HOME/.awarenessclaw/`
- Windows：`%USERPROFILE%\.awarenessclaw\`
- 通过 Node `os.homedir()` + `path.join()` 解析

## 二、单个 Mission 目录结构

```
~/.awarenessclaw/missions/mission-20260417-abc123/
├── mission.json          # 元数据（status / steps / timestamps）
├── plan.json             # Planner 输出的结构化计划（Plan B 用 JSON 不用 YAML，S1-T3 落地）
├── MEMORY.md             # 共享上下文，agent 读写
├── HEARTBEAT.md          # 心跳文件（30s 写一次）
├── artifacts/
│   ├── T1-init-nextjs.md
│   ├── T2-login-page.md
│   └── ...
└── logs/
    ├── T1-run.log        # 对应 step 的 Gateway 原始事件流
    ├── T2-run.log
    └── orchestrator.log  # 主循环的决策日志
```

> **2026-04-17 更新**：方案 B（TaskFlow + streaming）把 `plan.md`（YAML）改成 `plan.json`。Planner 输出 JSON，更易被 TypeScript / JSON Schema 校验。人类仍可直接阅读 JSON（4 空格缩进）。Markdown 形式的 plan 由 UI 层 PlanPreview 根据 plan.json 渲染，不作为权威源。

## 三、文件 Schema

### 3.1 `mission.json`

**schema**：

```jsonc
{
  "id": "mission-20260417-abc123",
  "version": 1,                          // schema 版本，用于迁移
  "goal": "做一个博客系统，用 Next.js...",
  "status": "running",                   // planning | running | paused | done | failed
  "createdAt": "2026-04-17T02:30:00Z",
  "startedAt": "2026-04-17T02:30:15Z",
  "completedAt": null,
  "plannerAgentId": "main",              // 哪个 agent 做的规划
  "rootWorkDir": "/Users/xx/projects/blog",  // agent 执行时的 cwd
  "steps": [
    {
      "id": "T1",
      "agentId": "coder",
      "agentName": "Dev",
      "agentEmoji": "💻",
      "role": "Developer",
      "title": "Initialize Next.js project",
      "deliverable": "markdown documenting scaffold",
      "depends_on": [],
      "status": "done",                  // waiting | running | retrying | done | failed | skipped
      "attempts": 1,
      "sessionKey": "agent:coder:subagent:uuid-xxx",
      "runId": "run-abc",
      "startedAt": "2026-04-17T02:30:15Z",
      "completedAt": "2026-04-17T02:33:42Z",
      "artifactPath": "artifacts/T1-init-nextjs.md",
      "errorCode": null,                 // network_error | agent_crash | permission_denied | ...
      "errorMessage": null
    }
  ],
  "currentStepId": "T2",                 // 下一个该跑的 step
  "lastEvent": {
    "at": "2026-04-17T02:40:05Z",
    "type": "step.progress",
    "stepId": "T2",
    "payload": "Installing next-auth..."
  }
}
```

**原子写规则**：
- 所有修改都走 `writeFileAtomic(path, JSON.stringify(data, null, 2))`
  - 实现：`fs.writeFile(path + '.tmp', content) → fs.rename(path + '.tmp', path)`
  - 保证 crash 时要么旧内容完整、要么新内容完整，不会半写
- 并发控制：一个 mission 目录同时只有一个 Runner 持有 `orchestrator.pid` 所有权（S2 实现）

**schema 版本迁移**：
- `version: 1` 是 S1-S3 版
- 如果读到 `version` 大于代码支持的版本：提示"该 mission 由更新版本创建，请升级 OCT-Agent"
- 如果读到更旧 version：走 `migrateMission(data, fromVersion)` 函数

### 3.2 `plan.md`

Planner 输出的人类可读版本 + JSON 数据。

```markdown
# Mission Plan: 做一个博客系统

> Generated by **main** (Claw) at 2026-04-17 02:30:00

## Summary

使用 Next.js 14 App Router + TypeScript + Tailwind + Prisma + NextAuth，
分 5 步完成：初始化 → 登录 → 文章 CRUD → 评论 → 测试。

## Subtasks

### T1 · Initialize Next.js project
- **Agent**: coder (Dev) · Developer
- **Deliverable**: markdown documenting scaffold
- **Depends on**: —
- **Estimated**: 5 min

### T2 · Implement login with NextAuth
- **Agent**: coder · Developer
- **Deliverable**: markdown documenting /app/login + auth config
- **Depends on**: T1
- **Estimated**: 10 min

...

---

<!--
JSON_PLAN_BEGIN
{
  "subtasks": [
    { "id": "T1", "agentId": "coder", "role": "Developer",
      "title": "...", "deliverable": "...", "depends_on": [] },
    ...
  ]
}
JSON_PLAN_END
-->
```

**规则**：
- 人类可读 markdown 在前，JSON 在 HTML 注释里供机器解析
- `JSON_PLAN_BEGIN` / `JSON_PLAN_END` 之间是严格 JSON，不允许注释
- 如果 JSON 和 markdown 冲突，**以 JSON 为准**（markdown 是 human-friendly 视图）

### 3.3 `MEMORY.md`

共享上下文。所有 step 读，done 时 append。

```markdown
# Mission Memory · 做一个博客系统

> Shared context across all agents in this mission.
> Each step appends a brief summary when done.

## Decisions made

- **技术栈**: Next.js 14 App Router + TypeScript + Tailwind + Prisma + NextAuth (by Planner at T0)
- **数据库**: 先用 SQLite 开发，prod 切 Postgres (by T1 at 2026-04-17 02:33)

## Files created

- `/app/page.tsx` (T1)
- `/app/login/page.tsx` (T2)
- `/app/api/auth/[...nextauth]/route.ts` (T2)

## Open questions

- 评论系统要支持 markdown 吗？— 等 T4 决定

## Gotchas

- **T1**: 用 `create-next-app` 的 `--use-pnpm` 比 `--use-npm` 快 3x
- **T2**: NextAuth v5 的 session provider 必须在 server component 里调 `auth()` 不是 `useSession()`
```

**规则**：
- `#` 一级标题 = mission name
- `##` 二级标题 = 分类（Decisions / Files / Questions / Gotchas / ...）
- 每条末尾标注 `(by <stepId>)`
- 新 step 追加时**只 append 不修改**（避免多 Runner 并发写冲突）
- 冲突时：最后一个 append 的为准（不做合并）

### 3.4 `HEARTBEAT.md`

Runner 每 30s 写一次，用于 crash 恢复定位。

```markdown
# Mission Heartbeat

- **Runner PID**: 12345
- **Last beat**: 2026-04-17T02:40:05Z
- **Current step**: T2
- **Step started at**: 2026-04-17T02:36:30Z
- **Step elapsed**: 3m 35s
- **Last event**: step.progress — Installing next-auth
- **Gateway session**: agent:coder:subagent:uuid-xxx
```

**用途**：
- Electron 重启时读 HEARTBEAT，如果 `Last beat` 在 2 分钟前 → 判定 Runner 死了 → 可以 resume
- 如果 `Last beat` 在 30s 内 → 认为 Runner 还活着，不抢夺

**原子写**：和 mission.json 一样走 writeFileAtomic。

### 3.5 `artifacts/T{n}-<slug>.md`

单个 step 的 deliverable。

```markdown
---
stepId: T1
agentId: coder
createdAt: 2026-04-17T02:33:42Z
durationSeconds: 207
---

# T1 · Initialize Next.js project

## What I did

运行 `pnpm create next-app blog --ts --tailwind --app --src-dir=false --import-alias="@/*"`，
然后 `pnpm add prisma next-auth @auth/prisma-adapter`。

## Key files

- `package.json` — 依赖清单
- `tsconfig.json` — 严格 TS
- `tailwind.config.ts` — Tailwind v3
- `prisma/schema.prisma` — 空 schema 占位

## Commands run

```bash
pnpm create next-app blog --ts --tailwind --app
cd blog && pnpm add prisma next-auth @auth/prisma-adapter
pnpm dlx prisma init --datasource-provider sqlite
```

## For the next agent

- 项目根目录在 `./blog/`，后续所有 path 相对这里
- SQLite 配置在 `prisma/schema.prisma`，datasource url = `file:./dev.db`
- 还没初始化 auth table schema，T2 的登录实现需要自己加
```

**规则**：
- YAML frontmatter 必选（`stepId` / `agentId` / `createdAt` / `durationSeconds`）
- `# Tx · <Title>` 一级标题和 step.title 一致
- **"For the next agent"** 段落是下一个 step 读 context 的重点，必须写
- 文件名 `T{n}-<slug>.md`，slug = step.title 小写去空格只保留字母数字短横线

### 3.6 `logs/T{n}-run.log`

Gateway 原始事件流 + orchestrator 决策。

```
[2026-04-17T02:30:15Z] [orchestrator] step T1 starting
[2026-04-17T02:30:15Z] [orchestrator] spawn: /subagents spawn coder "..."
[2026-04-17T02:30:16Z] [gateway] event:agent stream=lifecycle phase=start sessionKey=agent:coder:subagent:uuid
[2026-04-17T02:30:17Z] [gateway] event:chat state=delta role=assistant content="I'll initialize..."
[2026-04-17T02:30:25Z] [gateway] event:tool tool=exec args={"cmd":"pnpm create ..."}
[2026-04-17T02:33:40Z] [gateway] event:chat state=final role=assistant content="Done. Project at ./blog/"
[2026-04-17T02:33:41Z] [orchestrator] lifecycle end received, writing artifact
[2026-04-17T02:33:42Z] [orchestrator] step T1 done, elapsed 207s
```

**规则**：
- 每行 ISO 8601 时间 + source + message
- 大小控制：每条 log 单行 ≤ 4KB，超过截断
- 滚动：单个 T{n}-run.log ≤ 50 MB，超过自动 rotate 为 T{n}-run.log.1
- 保留策略（S3+）：mission done 30 天后压缩为 `T{n}-run.log.gz`

### 3.7 `logs/orchestrator.log`

Runner 主循环的全局日志。

```
[2026-04-17T02:30:00Z] mission-20260417-abc123 created
[2026-04-17T02:30:01Z] planner invoked: agentId=main
[2026-04-17T02:30:14Z] plan parsed: 5 subtasks
[2026-04-17T02:30:15Z] starting T1 → T2 → T3 → T4 → T5 sequential
[2026-04-17T02:33:42Z] T1 done
[2026-04-17T02:33:42Z] starting T2
```

## 四、并发 & 锁

### S1 单 Runner 约束

S1 阶段只有一个 Electron main 进程跑 Runner，不需要跨进程锁。

### S2 daemon 化后的锁

- Runner daemon 启动时：检查 `~/.awarenessclaw/orchestrator.pid`
  - 文件存在且 PID 还活着 → 报错退出（已有 daemon 在跑）
  - 文件存在但 PID 死了 → 抢占，写自己的 PID
  - 文件不存在 → 直接写
- 每个 mission 目录的锁：通过 `mission.json` 的 `runnerId` 字段 + HEARTBEAT
  - Runner 开始跑时写 `runnerId: <pid>`
  - 其他 Runner 看到 `runnerId != self` 且 HEARTBEAT 活着 → 跳过不抢
  - 看到 HEARTBEAT 死了（> 2min no beat）→ 抢占

## 五、清理策略

### S1 阶段

- 不自动清理，留给用户手动 `rm -rf ~/.awarenessclaw/missions/<id>`
- UI 提供 "删除 mission" 按钮，confirm 后删目录

### S3+ 阶段

- `mission.status === 'done'` 且 `completedAt` > 30 天 → 自动归档为 `missions/archive/<id>.tar.gz`
- `mission.status === 'failed'` 且 `completedAt` > 7 天 → 同样归档
- 归档只保留 `mission.json` + `plan.md` + `artifacts/*.md`，`logs/` 和 `HEARTBEAT.md` 删掉

## 六、隐私与安全

- `~/.awarenessclaw/` 权限 `0700`（只本用户可读写）
- `mission.json` / `plan.md` **不允许包含 API key / secrets**
  - Planner / Worker prompt 里要过一遍 secret 脱敏
  - 发现 `sk-...` / `ghp_...` / `api_key=...` 立即 redact
- 日志文件同样做 secret 脱敏（log rotation 前过一遍 filter）

## 七、跨平台注意点

- macOS / Linux 路径分隔符 `/`，Windows `\`
- 所有代码用 `path.join()` / `path.resolve()`，不用字符串拼接
- Windows 的 `~` 展开要用 `os.homedir()`，不是 shell 展开
- 文件名允许字符：`A-Za-z0-9._-`，禁用 `<>:"|?*`（Windows 保留字符）
